Sinapsis v4.3 — Dream Cycle

Higiene de indice inspirada en AutoDream de Anthropic

Que es el Dream Cycle

El Dream Cycle es un proceso de mantenimiento autonomo para el indice de instincts de Sinapsis. Ejecuta un analisis completo del fichero _instincts-index.json sin consumir un solo token de LLM: todo es Node.js determinista.

Se inspira en el concepto de AutoDream de Anthropic: un ciclo de "sueno" donde el sistema revisa su propia memoria, identifica problemas y aplica correcciones conservadoras de forma autonoma.

Principios clave:
  • 0 tokens de LLM — solo logica determinista en Node.js
  • 5 modulos independientes, cada uno con responsabilidad unica
  • Auto-acciones conservadoras (solo archiva borradores inactivos de 90+ dias)
  • Reporte interactivo con acciones manuales M/A/K para hallazgos no triviales
  • Produce un Health Score de 0 a 100 como metrica de salud del indice

Los 5 Modulos

1. Duplicados

Detecta pares de instincts cuyo contenido es demasiado similar para coexistir. Utiliza el coeficiente de Jaccard con umbral de 0.80 sobre tokens normalizados, ademas de comprobar solapamiento en los arrays de triggers.

Que detecta

  • Instincts con ≥80% de tokens compartidos en su contenido
  • Instincts con triggers identicos o subconjuntos del otro
  • Instincts fusionables donde uno es subconjunto semantico del otro

Ejemplo de hallazgo

{
  "type": "duplicate",
  "pair": ["supabase-rls-always", "supabase-rls-mandatory"],
  "jaccard": 0.87,
  "triggerOverlap": 3,
  "suggestion": "Merge into single instinct"
}

2. Contradicciones

Busca instincts que se contradicen entre si escaneando 7 pares de palabras opuestas en ingles y espanol. Si dos instincts activos contienen terminos opuestos en el mismo dominio, el modulo los marca como conflicto.

Pares opuestos detectados (EN + ES)

AfirmativoNegativo
always / siemprenever / nunca
require / requerirforbid / prohibir
enable / activardisable / desactivar
allow / permitirblock / bloquear
must / debemust not / no debe
include / incluirexclude / excluir
use / usaravoid / evitar

Ejemplo de hallazgo

{
  "type": "contradiction",
  "pair": ["use-docker-always", "avoid-docker-vps"],
  "opposites": ["always/use", "avoid"],
  "domain": "docker",
  "suggestion": "Resolve conflict — one must be archived"
}

3. Staleness

Clasifica cada instinct en cuatro categorias de frescura segun su actividad reciente, fecha de creacion y numero de ocurrencias (occurrences).

CategoriaCriterioAccion
fresh Ultima activacion < 30 dias Ninguna
stale Ultima activacion 30-89 dias Se reporta, accion manual
archive_candidate Ultima activacion ≥ 90 dias Se reporta. Auto-archivo si cumple regla
never_activated 0 ocurrencias desde su creacion Se reporta, accion manual

Regla de auto-archivo

Solo se auto-archivan instincts que cumplan las tres condiciones: confidence === "draft" + occurrences === 0 + edad ≥ 90 dias. Todo lo demas requiere decision humana.

4. Validacion de Triggers

Analiza los arrays de triggers de cada instinct buscando patrones problematicos que podrian causar activaciones incorrectas o excesivas.

Que detecta

  • Regex invalidas — triggers con sintaxis regex rota que crashearian el activator
  • Overly broad — triggers demasiado genericos (ej. file, code, error) que matchearian en casi toda sesion
  • Cross-domain — triggers de un dominio asignados a un instinct de otro dominio (ej. trigger stripe en un instinct de Supabase)

Ejemplo de hallazgo

{
  "type": "trigger_issue",
  "instinct": "prisma-json-parse",
  "trigger": ".*",
  "issue": "overly_broad",
  "suggestion": "Replace with specific trigger: prisma, json, schema"
}

5. Metricas de Salud

Calcula un Health Score de 0 a 100 que resume la calidad general del indice. La puntuacion se deriva de penalizaciones sobre un baseline perfecto.

Formula

score = 100 - (duplicates * 8) - (contradictions * 12) - (stale * 2) - (archive_candidates * 3) - (trigger_issues * 5)

El resultado se fija en el rango [0, 100] (nunca negativo).

FactorPenalizacionRazon
Cada par duplicado-8Ruido y confusion en activacion
Cada contradiccion-12Instrucciones conflictivas, mayor impacto
Cada instinct stale-2Peso muerto ligero
Cada archive_candidate-3Peso muerto significativo
Cada trigger issue-5Activaciones incorrectas

Como usar /dream

El comando /dream ejecuta el Dream Cycle completo y presenta un reporte interactivo.

Paso 1: Ejecutar el comando

/dream

Sinapsis lee _instincts-index.json, ejecuta los 5 modulos secuencialmente y genera el reporte en consola.

Paso 2: Revisar el reporte

El reporte muestra primero el Health Score y luego cada hallazgo agrupado por modulo. Cada hallazgo que requiere decision humana presenta tres acciones:

  • M Merge — Fusionar dos instincts en uno (para duplicados)
  • A Archive — Mover el instinct al archivo (_archived/)
  • K Keep — Mantener sin cambios (descarta el hallazgo)

Paso 3: Aplicar decisiones

El sistema aplica las decisiones de forma atomica: primero todas las fusiones, luego todos los archivados. Si hay un error, se revierte al estado previo.

Paso 4: Verificar resultado

Tras aplicar los cambios, /dream re-ejecuta el scoring y muestra el nuevo Health Score comparado con el anterior.

Auto-acciones

El Dream Cycle aplica una unica auto-accion, disenada para ser lo mas conservadora posible:

Auto-archivo de borradores inactivos

Un instinct se auto-archiva solo si cumple las tres condiciones simultaneamente:

  1. confidence === "draft" — Nunca se ha promovido a confirmed o permanent
  2. occurrences === 0 — Nunca ha sido activado por el instinct-activator
  3. Edad ≥ 90 dias — Lleva al menos 90 dias en el indice sin actividad

La logica es simple: si un instinct fue creado como borrador, nunca se activo en 90 dias y nunca fue promovido manualmente, es casi seguro que no aporta valor al indice. Archivarlo libera ruido sin riesgo de perder conocimiento validado.

Los instincts archivados no se eliminan — se mueven a _archived/ y pueden recuperarse con /instinct-import si fuera necesario.

Por que solo esta regla? Cualquier otra auto-accion (fusionar duplicados, resolver contradicciones, archivar instincts con ocurrencias) requiere juicio semantico que solo el operador puede hacer. El Dream Cycle es determinista: no adivina, no infiere, no usa LLM.

Ficheros

Ficheros leidos y escritos por el Dream Cycle:

FicheroOperacionDescripcion
_instincts-index.json Lectura + Escritura Indice principal de instincts. Se lee para analisis y se escribe si hay auto-archivados o merges
_archived/ Escritura Directorio donde se mueven los instincts archivados (auto o manual)
_dream-report.json Escritura Ultimo reporte completo del Dream Cycle en formato JSON para consumo programatico
_dream-log.jsonl Append Log historico de ejecuciones. Cada linea es un JSON con timestamp, score y resumen de acciones
_knowledge-index.md Lectura Se lee como contexto opcional si existe (producido por el linting diario Karpathy)
config.json No usado Eliminado en v4.0 — el Dream Cycle no tiene configuracion externa

Integracion

El Dream Cycle no es un hook. No se ejecuta automaticamente al abrir o cerrar sesion. Es un proceso independiente que se invoca de dos formas:

On-demand

El operador ejecuta /dream cuando quiere revisar la salud del indice. Tipicamente despues de una sesion intensa de aprendizaje o cuando el /instinct-status muestra demasiados instincts en draft.

Programado (scheduled task)

Puede configurarse como tarea programada via el sistema de scheduled tasks de Claude Code. La frecuencia recomendada depende del ritmo de aprendizaje:

Ritmo de usoFrecuencia recomendada
Uso diario intenso (5+ sesiones/dia)Diario
Uso moderado (1-3 sesiones/dia)Semanal
Uso esporadicoQuincenal o manual

Pipeline de Sinapsis

El Dream Cycle complementa el pipeline existente sin interferir: 

Session hooks (v4.0+):
  _session-learner.sh  -->  observe  -->  propose
  _instinct-activator.sh  -->  match triggers  -->  inject systemMessage

Mantenimiento (v4.2+):
  sinapsis-linting (scheduled)  -->  _knowledge-index.md
  /analyze-observations  -->  reads knowledge-index

Higiene (v4.3):
  /dream  -->  5 modulos  -->  reporte interactivo  -->  auto-acciones conservadoras

Instalacion

El Dream Cycle se instala automaticamente con install.sh de Sinapsis. No requiere pasos adicionales. 

# Instalacion completa de Sinapsis (incluye Dream Cycle)
curl -fsSL https://raw.githubusercontent.com/Luispitik/sinapsis-3.2/main/install.sh | bash

El instalador registra el comando /dream en el directorio de commands del proyecto y copia el modulo Node.js _dream-cycle.js al directorio core.

Verificar la instalacion

# Verificar que /dream esta disponible
ls .claude/commands/dream.md

# Ejecutar un Dream Cycle de prueba
/dream
Requisitos: Node.js 18+ (para las funciones de analisis determinista). No requiere dependencias externas — todo el codigo usa modulos nativos de Node.js.
