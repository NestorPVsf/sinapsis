Dream Cycle

Higiene de indice inspirada en AutoDream de Anthropic. Un proceso on-demand que analiza _instincts-index.json con 5 modulos deterministas (0 tokens LLM) y produce un reporte interactivo con Health Score. Se lanza con /dream o como scheduled task semanal.

Los 5 modulos

1. Duplicados

Detecta pares de instincts cuyo contenido es demasiado similar para coexistir. Utiliza el coeficiente de Jaccard con umbral de 0.80 sobre tokens normalizados, ademas de comprobar solapamiento en los arrays de triggers.

  • Instincts con 80%+ de tokens compartidos en su contenido
  • Instincts con triggers identicos o subconjuntos del otro
  • Instincts fusionables donde uno es subconjunto semantico del otro

2. Contradicciones

Busca instincts que se contradicen entre si escaneando 7 pares de palabras opuestas en ingles y espanol. Si dos instincts activos contienen terminos opuestos en el mismo dominio, el modulo los marca como conflicto.

  • always/siempre vs never/nunca
  • require/requerir vs forbid/prohibir
  • enable/activar vs disable/desactivar
  • allow/permitir vs block/bloquear
  • must/debe vs must not/no debe
  • include/incluir vs exclude/excluir
  • use/usar vs avoid/evitar

3. Staleness

Clasifica cada instinct por frescura segun actividad reciente, fecha de creacion y ocurrencias. Auto-archivo conservador: solo borradores inactivos 90+ dias.

  • fresh — Ultima activacion < 30 dias. Sin accion.
  • stale — Ultima activacion 30-89 dias. Se reporta, accion manual.
  • archive_candidate — Ultima activacion 90+ dias. Auto-archivo si: draft + 0 occurrences + 90+ dias.
  • never_activated — 0 ocurrencias desde creacion. Se reporta, accion manual.

4. Validacion de Triggers

Analiza los arrays de triggers de cada instinct buscando patrones problematicos que podrian causar activaciones incorrectas o excesivas.

  • Regex invalidas — triggers con sintaxis regex rota que crashearian el activator
  • Overly broad — triggers demasiado genericos (ej. file, code, error) que matchearian en casi toda sesion
  • Cross-domain — triggers de un dominio asignados a un instinct de otro dominio

5. Health Score (0-100)

Metrica unica que resume la calidad general del indice. Se deriva de penalizaciones sobre un baseline perfecto.

score = 100 - (duplicates * 8) - (contradictions * 12) - (stale * 2) - (archive_candidates * 3) - (trigger_issues * 5)

Auto-archivo

El Dream Cycle aplica una unica auto-accion conservadora. Un instinct se auto-archiva solo si cumple las tres condiciones simultaneamente:

confidence === "draft" + occurrences === 0 + edad >= 90 dias

Los instincts archivados se mueven a _archived/ y pueden recuperarse con /instinct-import. Todo lo demas requiere decision humana via el reporte interactivo (opciones M/A/K).

Confidence Decay

Sistema de degradacion automatica de confianza para mantener la higiene del indice. Los instincts que dejan de activarse pierden nivel progresivamente. Parte de la infraestructura de aprendizaje mantenida desde v4.4.

NivelRegla de decayResultado
confirmed Inactivo 60 dias sin activacion Demotido a draft
draft Inactivo 90 dias sin activacion Archivado automaticamente
permanent Nunca decae Siempre activo, sin degradacion

Como interactua con el Dream Cycle

Confidence Decay es el mecanismo de fondo que degrada niveles. El Dream Cycle es el proceso que detecta y reporta estos cambios. Juntos forman la capa de higiene del indice: Decay demota silenciosamente, Dream detecta y limpia explicitamente. Un instinct confirmed que lleva 60 dias sin activarse sera demotido a draft; si pasan otros 30 dias sin actividad, el Dream Cycle lo marcara como archive_candidate.

De cero a sistema completo

Sinapsis se instala con 3 instincts de ejemplo y 3 reglas basicas. El objetivo es que en 3-4 semanas tengas un sistema personalizado con 20-40 instincts de tu stack, reglas de tu equipo y skills propias. Asi es como se construye:

Dia 0 — Instalacion

0

Instalar Sinapsis

Clona el repo y ejecuta el installer. Configura hooks en settings.json, copia scripts, crea archivos iniciales con 3 instincts de ejemplo.

git clone https://github.com/Luispitik/sinapsis && cd sinapsis && bash install.sh

Resultado: 4 hooks activos (observe, passive-activator, instinct-activator, session-learner), 7 comandos disponibles, 3 instincts starter, 3 reglas pasivas basicas.

Semana 1 — Observar y sembrar

Trabaja y cierra bien cada sesion. Sinapsis acumula datos y el contexto se pasa de una sesion a la siguiente.

1

Trabaja normalmente (3-5 sesiones)

Cada tool call se registra en observations.jsonl automaticamente (async, 0 tokens). Al cerrar cada sesion, session-learner analiza las ultimas 1000 lineas y detecta patrones: errores que resolviste, archivos que editaste varias veces, secuencias de tools que repetiste.

Al final de cada dia, ejecuta /eod. Desde v4.2.2, /eod es multi-proyecto: ejecuta _eod-gather.sh que detecta TODOS los proyectos trabajados hoy (no solo el actual) via homunculus, agrega datos de git de cada root, y genera un resumen consolidado. Se guarda en _daily-summaries/.

Cuando abras la siguiente sesion, _project-context.sh inyecta ese resumen + el context.md tecnico automaticamente. Claude te saluda con lo que hiciste ayer y lo que toca hoy. No pierdes el hilo entre sesiones.

Opcionalmente, usa /session-end si quieres documentar sesiones intermedias (no solo al final del dia) con detalle de conversacion.

2

Anade 5-10 instincts manuales de tu stack

Piensa: que errores cometes siempre? Que gotchas de tu framework te muerden cada vez? Edita _instincts-index.json y anadelos. Ejemplos por stack:

Next.js: EPERM en .next/cache, locale prefix siempre "always", await en streamText()
Supabase: RLS obligatorio, split queries para evitar PGRST200, 3 gate points de auth
Stripe: customerId en modelo User, webhooks en modo test vs live
Django: makemigrations antes de migrate, CORS en production, SECRET_KEY no hardcodeado
React: useEffect cleanup, key en listas, memo para componentes pesados

3

Anade 3-5 reglas pasivas de tu equipo

Edita _passive-rules.json con guardrails que apliquen a todos tus proyectos. Estas se inyectan SIEMPRE que el trigger matchee, sin importar el proyecto:

Seguridad: "Toda API route debe validar JWT", "Nunca hardcodear URLs de produccion"
Calidad: "Tests antes de commit", "Conventional commits en ingles"
Stack: "Siempre Supabase con RLS", "Deploy siempre en Vercel"

Semana 2 — Analizar y confirmar

Ya tienes datos. Ahora extraes conocimiento.

4

Ejecuta /instinct-status

Mira que ha pasado: cuantas veces se han activado tus instincts manuales (occurrences), si hay drafts auto-detectados por session-learner, y si alguno ya se auto-promovio a confirmed (5+ matches).

/instinct-status
5

Ejecuta /analyze-session

Analisis profundo de la sesion: revisa observaciones, detecta patrones semanticos (errores, correcciones, workflows, preferencias), y propone instincts accionables. Mucho mas potente que revisar proposals crudas del regex.

/analyze-session

Acepta los que sean utiles. Descarta los que no. Cada aceptado entra como confirmed y se inyecta automaticamente a partir de ahora.

6

Revisa /passive-status

Mira que reglas se activan mas (esas son utiles) y cuales nunca se activan (quizas el trigger es demasiado restrictivo o ya no aplican). Ajusta o elimina las que no funcionen.

/passive-status

Semana 3+ — Evolucionar

Tu sistema ya tiene 15-25 instincts. Es hora de estructurar el conocimiento.

7

Ejecuta /evolve

Clusteriza instincts por dominio. Si tienes 5 instincts sobre Supabase, /evolve propone crear una skill "supabase-pro" que los agrupe. Si tienes 3 sobre seguridad, propone una regla pasiva compuesta. El conocimiento pasa de atomico (instincts sueltos) a estructurado (skills y reglas).

/evolve
8

Ejecuta /dream para higiene del indice

Lanza el Dream Cycle para limpiar duplicados, detectar contradicciones, identificar instincts obsoletos y validar triggers. Te da un Health Score de 0 a 100 y propone acciones concretas.

/dream
9

Promueve los mejores a permanentes

Los instincts que llevan semanas activandose y nunca te han dado una sugerencia incorrecta son candidatos a /promote. Un instinct permanent tiene prioridad maxima, se aplica siempre y nunca decae.

/promote supabase-rls-required
10

El ciclo continuo

A partir de aqui, el sistema crece solo. Cada sesion genera datos, session-learner detecta patrones, instinct-activator trackea lo que funciona, y tu haces /analyze-session cada semana o dos para refinar. El linting diario detecta gaps automaticamente. El Dream Cycle mantiene el indice limpio. Confidence Decay degrada lo inactivo.

Resultado tipico en 1 mes: 25-40 instincts, 8-12 reglas pasivas, 2-3 skills evolucionadas. Tu agente ya sabe como trabajas.

Preguntas frecuentes

Cuantos tokens consume?
Zero en hooks (observe, activator, session-learner son Node.js/Python, no LLM). Solo consume tokens al ejecutar /analyze-session, /evolve o /dream. Overhead total: 6,915 tokens/sesion.
Funciona con otros agentes ademas de Claude Code?
Actualmente solo Claude Code (los hooks son especificos de settings.json). La arquitectura es portable, pero requiere adaptacion de hooks para otros agentes.
Puedo anadir instincts manualmente?
Si. Edita _instincts-index.json directamente. Anade un objeto con id, trigger_pattern (regex), inject (texto), level, scope, domain. El activator lo usara en la siguiente sesion.
Que pasa si borro observations.jsonl?
Se crea uno nuevo automaticamente. Pierdes el historico de ese proyecto, pero los instincts ya generados se mantienen en el indice.
Como hago backup?
El installer ofrece backup automatico al actualizar. Manualmente: copia _instincts-index.json, _session-learner.sh, _instinct-activator.sh a una carpeta segura. Usa /instinct-export para exportar instincts a YAML.
Es seguro hacer publico el repo?
Si, siempre que no commits _operator-state.json, _passive-rules.json ni _instincts-index.json. Estos contienen datos de negocio personales. El .gitignore los excluye automaticamente.
Que paso con review-army, cso-audit e investigate-pro?
Se separaron en v4.3.2. Sinapsis se centra solo en aprendizaje. Las herramientas de ingenieria estan archivadas en ~/.claude/skills/_archived/sinapsis-gstack/ con RECOVERY.md y se moveran a un repo independiente.
